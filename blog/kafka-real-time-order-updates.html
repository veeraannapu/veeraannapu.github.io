How Kafka Powers Real-Time Order Updates at Scale

VA
Veera Annapureddy
·
May 2025
8 min read

When a customer places an order on a large e-commerce platform, a lot happens in the next few seconds that they never see. Their order confirmation email fires. Their order history updates. The fulfillment system picks it up. The analytics pipeline records the event. A dozen downstream systems all need to know — right now — that something happened.

The naive approach is to call each of those systems directly. One service, twelve API calls. That works fine until it doesn't: one downstream service is slow, and now your checkout response time triples. One is down, and now orders silently fail to reach fulfillment.

This is the problem Apache Kafka solves — and after building order event pipelines that process hundreds of thousands of messages a day, here's what I've learned about doing it right.

What Kafka Actually Is (and Isn't)

Kafka is often described as a message queue. That's not wrong, but it undersells it. The better mental model is a distributed, append-only log.

When your order service publishes an event to Kafka, it writes a record to a topic. That record stays there — for hours, days, or weeks depending on your retention config. Any number of consumers can read it independently, at their own pace, without affecting each other or the producer.

This is fundamentally different from a queue:

Queue model: Producer → [Queue] → Consumer A ↑ message deleted after consumed Kafka model: Producer → [Topic / Log] → Consumer Group A (email service) → Consumer Group B (fulfillment) → Consumer Group C (analytics) ↑ message retained, each group tracks its own offset

That distinction matters enormously in an order system. You don't want fulfillment consuming the same message as email notifications. You want both to get it.

The Order Event Model

Before you write a single line of Kafka config, get your event schema right. This is where most teams make their first mistake — they design events as database row dumps instead of meaningful domain events.

Bad event design

{
  "table": "orders",
  "operation": "UPDATE",
  "before": { "status": "PENDING" },
  "after":  { "status": "CONFIRMED" }
}

This is a database change event masquerading as a domain event. Consumers now need to understand your database schema and diff logic to figure out what happened.

Better event design

{
  "eventType":   "ORDER_CONFIRMED",
  "eventId":     "evt-8f3a2c1d",
  "occurredAt":  "2025-05-01T14:32:00Z",
  "orderId":     "ord-00492817",
  "customerId":  "cust-10034",
  "orderTotal":  142.50,
  "currency":    "USD",
  "itemCount":   3,
  "source":      "WEB_CHECKOUT"
}

The event says what happened in the business, not what changed in the database. Consumers subscribe to ORDER_CONFIRMED and know exactly what to do with it.

Event types in an order system

Model each lifecycle transition as its own event type:

ORDER_CREATED
ORDER_CONFIRMED
ORDER_PAYMENT_CAPTURED
ORDER_SUBMITTED_TO_FULFILLMENT
ORDER_SHIPPED
ORDER_DELIVERED
ORDER_CANCELLATION_REQUESTED
ORDER_CANCELLED
ORDER_RETURN_INITIATED
ORDER_REFUNDED

Partitioning: The Decision That's Hard to Undo

Kafka topics are divided into partitions. Messages within a partition are ordered. Messages across partitions are not. How you partition determines both your throughput ceiling and your ordering guarantees.

For order events, partition by order ID:

ProducerRecord<String, OrderEvent> record =
    new ProducerRecord<>(
        "order-events",   // topic
        order.getId(),    // partition key → orderId
        event             // value
    );

This guarantees all events for a given order land in the same partition, in the order they were produced. Your fulfillment consumer will always see ORDER_CONFIRMED before ORDER_SHIPPED for the same order. If you partition randomly, you lose that guarantee.

Tip — Partition Count

Each partition can be consumed by exactly one consumer in a group at a time. If you have 12 partitions and 6 consumers, each consumer handles 2 partitions. If you have 12 partitions and 20 consumers, 8 consumers sit idle. Start with a partition count you can scale into — it's painful to repartition a live topic.

Consumer Groups: Isolation Without Duplication

Every downstream system that needs order events should have its own consumer group:

order-events topic │ ├── consumer-group: email-service │ └── Sends order confirmation emails │ ├── consumer-group: fulfillment-service │ └── Routes orders to warehouse │ ├── consumer-group: analytics-pipeline │ └── Writes to data lake │ └── consumer-group: search-indexer └── Updates order search index

Each group maintains its own offset — its position in the log. The email service and fulfillment service are completely independent. One being slow or behind doesn't affect the other.

This is also what makes Kafka replay so powerful. If your analytics pipeline goes down for 6 hours, it comes back up, reads from where it left off, and catches up. No events were lost. No backfill job needed.

Handling Failures: At-Least-Once vs Exactly-Once

Here's the uncomfortable truth: Kafka delivers messages at-least-once by default. Your consumer will receive every message at least once, but under failure conditions — consumer crash, rebalance, network blip — it might receive the same message twice.

For order events, this is a real problem. You don't want to send two confirmation emails or submit an order to fulfillment twice.

Idempotent Consumers

Design your consumers so processing the same event twice has no additional effect:

@KafkaListener(topics = "order-events", groupId = "email-service")
public void handleOrderEvent(OrderEvent event) {
    if (emailRepository.existsByEventId(event.getEventId())) {
        log.info("Duplicate event, skipping: {}", event.getEventId());
        return;
    }
    emailService.sendConfirmation(event);
    emailRepository.markProcessed(event.getEventId());
}

Store the eventId and check it before processing. This is simple, reliable, and works across consumer restarts.

Offset Management: Don't Auto-Commit

The default Kafka consumer config auto-commits offsets on a timer. This creates a subtle bug:

⚠ The Auto-Commit Trap

Consumer reads message → Auto-commit fires → offset marked as processed → Consumer processes message → throws exception → Message was "committed" but never actually processed.

Disable auto-commit and commit manually after successful processing:

# application.yml
spring:
  kafka:
    consumer:
      enable-auto-commit: false
      auto-offset-reset: earliest
@KafkaListener(topics = "order-events")
public void consume(ConsumerRecord<String, OrderEvent> record,
                    Acknowledgment ack) {
    try {
        processEvent(record.value());
        ack.acknowledge(); // commit only after success
    } catch (RetryableException e) {
        // don't ack — message will be redelivered
        log.warn("Retryable error, will retry: {}", e.getMessage());
    } catch (FatalException e) {
        // send to dead letter topic, then ack to move on
        deadLetterProducer.send(record);
        ack.acknowledge();
    }
}

Dead Letter Topics

Not every message can be processed. A malformed event, a downstream outage, a bug in consumer logic — sometimes a message will fail repeatedly. If you let it sit at the head of the partition, it blocks all subsequent messages for that partition forever.

The solution is a dead letter topic (DLT):

@Bean
public DeadLetterPublishingRecoverer recoverer(KafkaTemplate<?, ?> template) {
    return new DeadLetterPublishingRecoverer(template,
        (record, ex) -> new TopicPartition(
            record.topic() + ".DLT",
            record.partition()
        )
    );
}

Monitor your DLT message count. A spike tells you something broke in your consumer or your upstream data.

What I'd Do Differently

  • Version your events from day one. Add a schemaVersion field even on v1. When you need to change the shape of an event in a live system, you'll thank yourself.
  • Don't put large payloads in Kafka. Put the order ID in the event and let consumers fetch details from the order service. Kafka is optimized for small, frequent messages — not large documents.
  • Schema Registry is worth it. Confluent Schema Registry with Avro or Protobuf enforces that producers and consumers agree on the event shape.
  • Test your consumer rebalancing. Spin up 4 consumers, kill 2, restart them. Watch what happens to lag and processing guarantees. Do this in staging, not production.

Summary

Kafka's power in an order system comes from one core idea: decoupling the producer from its consumers. Your order service publishes one event. Everything downstream consumes it independently, at its own pace, with its own retry and failure handling.

Getting this right means designing events as domain events (not database dumps), partitioning by order ID, giving each downstream system its own consumer group, committing offsets manually, building idempotent consumers, and monitoring consumer lag and DLT rates.

The investment pays off when one of your downstream services goes down at 2 AM and your order flow keeps humming — events are just waiting in the log, ready to be consumed when the service comes back up.
