Medallion Architecture Explained: Raw → Bronze → Silver → Gold

VA
Veera Annapureddy
·
May 2025
6 min read

If you've worked with a data lakehouse, you've probably heard the term "medallion architecture." It sounds more formal than it is. At its core it's a layering pattern — a way of organizing data in your lake so that every downstream consumer gets data that's been cleaned and validated to the right level for their use case.

After building and maintaining medallion pipelines that process order data at scale, here's a practical breakdown of what each layer actually does, why it exists, and the design decisions that matter.

The Problem It Solves

Without a structured approach, data lakes become data swamps. Raw data lands in S3 or ADLS, pipelines transform it in ad-hoc ways, and six months later nobody agrees on what the "correct" customer count is because three teams ran three different transformations.

The medallion pattern imposes a contract: data only flows forward through layers, and each layer has a clear definition of quality.

RAW
Source Systems → LakeExact copy of source data. Immutable. Never modified.
↓
BRONZE
Ingested + StructuredSchema applied, audit columns added, append-only.
↓
SILVER
Cleaned + ValidatedDeduplicated, validated, enriched. Source of truth.
↓
GOLD
Business-ReadyAggregated, purpose-built for dashboards, APIs, and ML.

Each layer is a contract. Bronze says: "this data arrived." Silver says: "this data is clean." Gold says: "this data is ready to answer a business question."

Raw Layer — Touch Nothing

The raw layer is exactly what it sounds like: the original data, untouched, as it came from the source system. If you're consuming from Kafka, raw is the Parquet files landed from your Kafka consumer before any transformation runs.

Rules for the raw layer:

  • Never modify raw data once written
  • Preserve the original format where possible
  • Add landing metadata: landed_at timestamp, source system, file path
  • Partition by ingestion date, not business date
spark \
  .readStream \
  .format("kafka") \
  .option("kafka.bootstrap.servers", KAFKA_BROKERS) \
  .option("subscribe", "order-events") \
  .load() \
  .select(
      col("value").cast("string").alias("raw_payload"),
      col("timestamp").alias("kafka_timestamp"),
      col("partition"),
      col("offset"),
      current_timestamp().alias("landed_at"),
  ) \
  .writeStream \
  .format("delta") \
  .outputMode("append") \
  .option("checkpointLocation", "/checkpoints/raw/orders") \
  .partitionBy("landed_at") \
  .start("/data/raw/order-events")
Why bother with raw?

Because sources change. Upstream teams rename fields, change data types, drop columns. When that happens and your pipeline breaks, raw is your time machine. You can replay from raw with the correct transformation logic without going back to the source.

Bronze Layer — Make It Readable

Bronze takes raw data and applies structure. The data is still unvalidated and may contain duplicates, nulls, and bad values — but it has a consistent schema and the audit columns that make every downstream layer traceable.

What happens in bronze: parse JSON into typed columns, apply the target schema, add standard audit columns (_ingested_at, _source_file, _batch_id), handle schema evolution, write as Delta Lake tables.

bronze_df = (
    spark.readStream
    .format("delta")
    .load("/data/raw/order-events")
    .select(
        from_json(col("raw_payload"), ORDER_SCHEMA).alias("data"),
        col("landed_at"),
        input_file_name().alias("_source_file"),
    )
    .select(
        "data.*",
        col("landed_at").alias("_ingested_at"),
        col("_source_file"),
        current_timestamp().alias("_bronze_created_at"),
    )
    .withColumn("occurred_at", col("occurredAt").cast(TimestampType()))
)

bronze_df.writeStream \
    .format("delta") \
    .outputMode("append") \
    .option("checkpointLocation", "/checkpoints/bronze/orders") \
    .option("mergeSchema", "true") \
    .partitionBy("eventType") \
    .toTable("lakehouse.bronze.order_events")

Bronze is append-only. You don't update or delete records at this layer. Every record that ever came through is here. Silver and gold handle correctness; bronze handles completeness.

Silver Layer — Make It Correct

Silver is where the real work happens. This is the layer that most data consumers wish existed and most data teams underinvest in.

What happens in silver: deduplication (the same event may arrive twice from Kafka), null handling and validation, enrichment (join with reference data), business rule application, type-corrected and consistent data.

def upsert_to_silver(batch_df, batch_id):
    # Filter valid records only
    valid = batch_df.filter(
        col("eventId").isNotNull() &
        col("orderId").isNotNull() &
        col("orderTotal").isNotNull() &
        (col("orderTotal") >= 0)
    )

    # Deduplicate within the batch (take latest by occurredAt)
    deduped = valid \
        .withColumn("row_num", row_number().over(
            Window.partitionBy("eventId")
                  .orderBy(col("occurred_at").desc())
        )) \
        .filter(col("row_num") == 1) \
        .drop("row_num")

    # Enrich with reference data
    enriched = deduped.join(
        spark.table("lakehouse.reference.customers"),
        on="customerId", how="left"
    )

    # MERGE into silver — handles late arrivals cleanly
    silver_table = DeltaTable.forName(spark, "lakehouse.silver.order_events")
    silver_table.alias("target").merge(
        enriched.alias("source"),
        "target.eventId = source.eventId"
    ) \
    .whenMatchedUpdateAll() \
    .whenNotMatchedInsertAll() \
    .execute()
Silver uses MERGE, not append

Late-arriving data is real — Kafka guarantees at-least-once delivery, and network issues mean events sometimes arrive out of order. Silver handles this gracefully with upsert semantics. Silver is the layer most consumers should read from.

Gold Layer — Make It Fast

Gold is purpose-built for specific business questions. It's pre-aggregated, pre-joined, and optimized for query performance.

# Gold: Daily order summary by channel
gold_daily = (
    spark.table("lakehouse.silver.order_events")
    .filter(col("eventType") == "ORDER_CONFIRMED")
    .groupBy(
        to_date(col("occurred_at")).alias("order_date"),
        col("source").alias("channel"),
    )
    .agg(
        count("*").alias("order_count"),
        sum("orderTotal").alias("gross_revenue"),
        avg("orderTotal").alias("avg_order_value"),
        countDistinct("customerId").alias("unique_customers"),
    )
    .withColumn("_gold_updated_at", current_timestamp())
)

gold_daily.write \
    .format("delta") \
    .mode("overwrite") \
    .partitionBy("order_date") \
    .saveAsTable("lakehouse.gold.daily_order_summary")

You might have multiple gold tables serving different audiences — a daily summary for the executive dashboard, an order history table for the customer-facing API, a fulfillment queue for warehouse operations. Each is optimized for its consumer. The query that the BI tool runs against gold should be trivial — a simple SELECT with no joins.

5 Common Mistakes

1
Skipping raw because "we control the source"You don't control the source as much as you think. Schema changes, upstream bugs, and production incidents happen. Keep raw.
2
Putting business logic in bronzeBronze should be mechanical transformation — parse and land. The moment you apply business rules in bronze, you've contaminated your audit layer with assumptions that will change.
3
Treating silver as a passthroughIf your silver layer is just bronze with a different name, you've missed the point. Silver needs deduplication, validation, and enrichment. If it doesn't, your gold will be wrong.
4
One gold table to rule them allA single wide gold table that tries to serve every consumer usually serves none well. Build narrowly scoped gold tables for each use case.
5
Not setting retention policies per layerRaw: long (1+ year, audit trail). Bronze: medium (90 days). Silver: medium-long (source of truth). Gold: short (rebuilt regularly, small).

Putting It All Together

Kafka (order-events) │ ▼ RAW (Delta) ├── Immutable landing zone ├── Full Kafka metadata preserved └── Partitioned by ingestion date │ ▼ BRONZE (Delta) ├── Typed schema applied ├── Audit columns added (_ingested_at, _source_file) ├── Append-only └── Partitioned by eventType │ ▼ SILVER (Delta) ├── Deduplicated on eventId ├── Validated (nulls, business rules) ├── Enriched with reference data ├── Upsert (handles late arrivals) └── Source of truth for analytics │ ▼ GOLD (Delta → Snowflake) ├── Pre-aggregated by business grain ├── Purpose-built per consumer ├── Exposed to BI / dashboards / APIs └── Rebuilt on schedule

The medallion pattern isn't magic. It's a discipline — a commitment to not letting data move forward until it meets the contract of each layer. That discipline is what separates a data lake people trust from one they work around.
