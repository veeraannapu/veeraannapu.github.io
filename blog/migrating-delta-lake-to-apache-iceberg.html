Migrating from Delta Lake to Apache Iceberg — What We Learned

VA
Veera Annapureddy
·
May 2025
10 min read

We didn't wake up one morning and decide to migrate. Like most infrastructure decisions, it started with a problem — a growing frustration with cross-engine compatibility that Delta Lake, in its earlier versions, didn't handle well. Our Databricks pipelines worked great. But the moment another team tried to query the same tables from a non-Spark engine, things got complicated.

This is an honest account of why we migrated, how we did it, what surprised us, and what I'd do differently with the benefit of hindsight.

Why We Started Looking at Iceberg

Delta Lake is excellent inside the Databricks ecosystem. ACID transactions, time travel, schema evolution — it does all of these well. But it has always been tightly coupled to Spark, and as our data ecosystem grew, that coupling became friction.

The specific pain points that pushed us toward Iceberg:

  • Cross-engine reads were painful. Other teams using Trino, Presto, or Athena to query our Delta tables had to jump through hoops — manifest files, connector configurations, version mismatches.
  • Metadata operations were slow at scale. On large tables, listing Delta transaction logs became a bottleneck. VACUUM and OPTIMIZE operations that should be routine started taking hours.
  • Vendor lock-in felt real. Delta Lake is open source but Databricks controls the roadmap. Iceberg is governed by the Apache Software Foundation — multiple vendors contribute, and no single company owns the format.
🔶 Delta Lake
  • Excellent Databricks integration
  • Simple transaction log (_delta_log)
  • Mature MERGE and DML support
  • Spark-native, great performance
  • Cross-engine support improving but limited
  • Databricks-led roadmap
🧊 Apache Iceberg
  • True open table format — engine agnostic
  • Rich metadata layer (manifests, snapshots)
  • Native Spark, Trino, Flink, Athena support
  • Hidden partitioning — no partition column needed in queries
  • Partition evolution without rewriting data
  • Apache Foundation governance

What "Migration" Actually Means

Before getting into the how, it's worth being clear about what we were actually doing. There is no magic "convert Delta to Iceberg" button. The data doesn't change — Parquet files are Parquet files. What changes is the metadata layer that sits on top: the transaction logs, manifests, and catalog entries that tell engines how to read those files.

A Delta table stores its metadata in a _delta_log/ folder full of JSON transaction logs. An Iceberg table stores its metadata in a hierarchy of manifest files, snapshot files, and a catalog entry. Same underlying Parquet data — completely different metadata structure.

This means a migration is fundamentally a metadata operation, which is both good news (you're not rewriting terabytes of data) and a source of risk (if the metadata migration goes wrong, queries break silently).

How We Approached It

Phase 1 — Inventory
Catalog every Delta table, classify by risk
We listed all Delta tables, noted their size, daily write volume, downstream consumers, and criticality. Low-traffic analytics tables migrated first. High-traffic production tables last.
Phase 2 — Shadow tables
Build Iceberg versions alongside Delta tables
For each table, we created an Iceberg version running in parallel. Both were written to simultaneously. Consumers could query either. This ran for 2 weeks per table before cutover.
Phase 3 — Validation
Row count checks, schema validation, sample diffs
Automated reconciliation jobs ran nightly comparing Delta vs Iceberg row counts, schema definitions, and sample record diffs. Any discrepancy blocked the cutover for that table.
Phase 4 — Cutover
Switch consumers, keep Delta as read-only backup
Consumer jobs were pointed at Iceberg tables. Delta versions stayed alive in read-only mode for 30 days as rollback insurance. No data was deleted during this window.
Phase 5 — Cleanup
Decommission Delta versions after 30 days
After 30 stable days on Iceberg, Delta versions were decommissioned. Storage reclaimed, pipelines simplified.

The Technical Migration

For tables where a full backfill was acceptable, we used Spark to read the Delta table and write to Iceberg directly:

# Read existing Delta table
delta_df = spark.read \
    .format("delta") \
    .load("s3://our-lake/bronze/order_events")

# Write as Iceberg to new location
delta_df.write \
    .format("iceberg") \
    .mode("overwrite") \
    .option("write.distribution-mode", "hash") \
    .saveAsTable("catalog.bronze.order_events_iceberg")

For large tables where a full backfill would be too slow, we used Iceberg's snapshot migration — registering existing Parquet files directly into an Iceberg catalog without moving them:

from pyspark.sql import SparkSession

# Register existing Parquet files as an Iceberg table
# This does NOT copy data — it just creates Iceberg metadata
# pointing at the same files Delta was using

spark.sql("""
    CALL catalog.system.add_files(
        table => 'catalog.bronze.order_events_iceberg',
        source_table => '`delta`.`s3://our-lake/bronze/order_events`'
    )
""")
Snapshot Migration

Iceberg's add_files procedure is the fastest path for large tables. It creates Iceberg metadata pointing at existing Parquet files — no data movement, no rewrite. The trade-off is that time travel history from Delta doesn't carry over — you start fresh in Iceberg from the migration point.

What Broke

I'd be lying if I said it went smoothly. Three things caught us:

1. Partition scheme differences

Our Delta tables were partitioned by event_date as a physical column — consumers passed WHERE event_date = '2025-01-01' in their queries. Iceberg's hidden partitioning means the partition column doesn't need to appear in the data. When we migrated, queries that referenced event_date in the partition definition but not as a data column started failing. We had to decide: re-add the column to the data, or update all consumers. We updated consumers — the right call, but more work than expected.

2. MERGE behaviour differences

Delta and Iceberg handle MERGE semantics slightly differently, particularly around how they handle matched rows with null keys. We had one pipeline that relied on an edge case in Delta's MERGE implementation. Moving to Iceberg exposed it.

-- This behaved differently in Iceberg vs Delta
-- when source.key IS NULL
MERGE INTO target t
USING source s ON t.key = s.key
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *

Always test your MERGE logic explicitly against edge cases during migration validation.

3. Catalog configuration complexity

Iceberg requires a catalog — REST, Glue, Hive Metastore, or Nessie. Delta uses the file system directly (_delta_log/ is self-contained). Getting Iceberg's catalog configuration consistent across Spark, Databricks, and the teams using Trino took more coordination than anticipated.

What Got Better

The migration pain was real, but so were the benefits once we were on the other side.

Cross-engine queries just worked. Trino could read our Iceberg tables with zero special configuration. The Iceberg REST catalog made this seamless — engines connected to one catalog, got consistent metadata, and queried the same underlying files.

Partition evolution saved us twice. Post-migration, we needed to change the partitioning on two tables — one from daily to hourly, one from event type to customer region. In Delta, this would have required a full table rewrite. In Iceberg, ALTER TABLE ... ADD PARTITION FIELD applies going forward without touching historical data.

-- Add a new partition field without rewriting existing data
ALTER TABLE catalog.silver.order_events
ADD PARTITION FIELD days(occurred_at);

-- Old partitions remain as-is
-- New data partitioned by day going forward

Metadata queries are faster. Iceberg's manifest files are designed for pruning — querying metadata to find which files contain relevant data is significantly faster than scanning Delta's transaction log at scale. On our largest table (~800M rows), partition pruning queries that took 40+ seconds in Delta run in under 8 seconds in Iceberg.

Was It Worth It?

For our use case — yes, clearly. The cross-engine compatibility alone justified the effort. But I want to be honest: if you're purely a Databricks shop with no other query engines and no plans to change that, Delta Lake is an excellent choice and migration isn't obviously worth the disruption.

Iceberg is the right choice when:

  • You have multiple query engines reading the same data
  • You need engine-agnostic open format guarantees
  • You're doing partition evolution frequently
  • You're building for long-term flexibility over short-term convenience

Delta Lake is still the right choice when:

  • Your entire stack is Databricks or Spark
  • You need the most mature DML support available today
  • Your team doesn't have bandwidth for catalog infrastructure work

If I Did It Again

Start with the catalog early. Don't think of catalog setup as a pre-migration step — it's the foundation everything else sits on. Get it right first.

Run shadow tables longer than you think you need to. Two weeks felt like a lot. It wasn't. An extra week of parallel running catches the edge cases that weekly batch jobs surface.

Write an explicit migration runbook per table. Each table had different consumers, different partition schemes, different SLAs. A single generic process doesn't work. Document the specific steps, validation queries, and rollback plan for each table before you start.

Test your MERGE logic explicitly. Don't assume it'll behave the same. Write test cases covering null keys, duplicate source rows, and matched+unmatched combinations before cutover.
