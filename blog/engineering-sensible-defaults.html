Sensible Defaults: The Engineering Habits That Changed How My Team Works

VA
Veera Annapureddy
·
May 2025
6 min read

"Sensible defaults" is one of those phrases that sounds obvious until you actually try to define it. Of course defaults should be sensible — who's arguing otherwise? But the phrase only started meaning something concrete to me when I started following it as a structured set of engineering practices, not just a vague principle.

This post is about what that looks like in practice — the specific habits, what they're for, and how a team adopts them without it feeling like a mandate from above.

The Crawl → Walk → Run Model

The framework I've found useful organises sensible defaults into three maturity stages. You don't try to do everything at once — you crawl first, build the foundation, then walk, then run. Each stage builds on the one before it.

🚶

The model acknowledges something important: not every team is in the same place. A team that's never done pair programming shouldn't try to simultaneously adopt trunk-based development, TDD, and automated deployments. Pick the crawl practices first, get them stable, then move on. Trying to skip stages is where adoption fails.

Crawl — Getting the Foundations Right

Small Stories

Breaking work into genuinely small stories is harder than it sounds and more valuable than most teams realise. A story that takes two weeks to deliver means two weeks before you get feedback. A story that takes two days means you can course-correct five times in the same period.

The discipline isn't just about estimation — it's about how you decompose problems. Small stories force you to think clearly about what the minimum valuable slice of work actually is. That thinking improves design, reveals dependencies early, and makes review much easier because there's less to hold in your head at once.

Pair Programming

Two engineers, one keyboard. Pairing is part of the Crawl stage because it's one of the highest-leverage things a team can adopt early. It spreads knowledge, catches bugs before they're committed, forces design conversations, and accelerates onboarding in a way that documentation never quite does.

It feels slow. The output — lines of code written per hour — looks lower than solo work. The quality of what's produced, and how well both engineers understand it, is usually significantly higher.

Team Ownership of Quality

Quality isn't a QA team's responsibility. It isn't a code reviewer's responsibility. It's everyone's, at every stage — when you write the code, when you review it, when you ship it. A team that internalises this stops treating code review as a gate and starts treating it as a conversation. The difference shows up in what gets caught and when.

Walk — Building Engineering Discipline

Trunk-Based Development

Everyone commits to a single shared branch — trunk or main — at least once a day. No long-lived feature branches sitting around for weeks accumulating merge conflicts. This sounds aggressive until you combine it with feature toggles, which let you merge incomplete work safely because it's hidden behind a flag until it's ready.

if (featureFlags.isEnabled("new-order-routing", customerId)) {
    return newOrderRouter.route(order);
} else {
    return legacyOrderRouter.route(order);
}

The discipline of committing often to trunk is what keeps codebases from developing the kind of long-running divergence that makes merging painful and deployment unpredictable.

Test Driven Development

Write the test before the code. This sounds backwards until you do it enough to understand what it actually produces: code that was designed to be testable from the start, tests that reflect the intended behaviour rather than the implementation, and a reliable safety net for every change that comes after.

TDD isn't about test coverage metrics. It's about thinking clearly about what a piece of code should do before you write it — and having a fast, automated way to verify that it still does that after every change.

Fast Automated Builds

A build that takes 45 minutes doesn't get run. A build that takes 4 minutes gets run on every commit. Speed isn't a vanity metric — it determines whether your automation is actually used as a feedback loop or just a ritual before deployment. Keep builds fast, keep them green, and treat a failing build as the team's top priority until it's fixed.

Run — Continuous Delivery

The Run stage is where the team starts deploying frequently, automatically, and with confidence. This requires everything from the Crawl and Walk stages to already be working — you can't automate your way to fast, reliable deployments if your tests are slow, your stories are large, or your quality is owned by a bottleneck.

🚀 Automated Deployments

Every merge to trunk triggers a pipeline that builds, tests, and deploys — without a human in the loop. The goal is to make deployment a non-event. Something that happens dozens of times a day without anyone thinking about it.

🔒 Build In Security

Security checks — dependency scanning, secret detection, SAST — run in the pipeline, not as a separate process before release. Security that's a separate gate becomes a bottleneck. Security that's built into every build becomes invisible and reliable.

🏭 Build for Production

Every build is production-ready. Not "mostly ready" or "close enough for staging." If a build passes the pipeline, it can go to production. This changes how you think about what goes into the pipeline — every check that matters for production needs to live there.

Why the Order Matters

The sequence — Crawl, Walk, Run — is deliberate. Teams that try to jump straight to automated deployments without small stories, pairing, or reliable tests end up automating chaos faster. The foundation has to be there first.

Adopting sensible defaults isn't a big-bang transformation. It's an incremental improvement to how a team works, where each stage creates the conditions for the next one to succeed. The teams I've seen do this well didn't announce a new process and mandate compliance. They picked one practice, made it the path of least resistance, let it become normal — then picked the next one.

Where to start

If you're adopting this for the first time: start with small stories and team ownership of quality. Those two practices alone change the texture of how a team works — and they make everything else in the journey easier to adopt.
