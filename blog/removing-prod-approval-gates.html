Why I Removed the PROD Manual Approval Gate — And What Replaced It

VA
Veera Annapureddy
·
May 2025
8 min read

Manual approval gates feel safe. Someone looks at what's about to go to production, clicks approve, and the deployment proceeds. There's a human in the loop. What could go wrong?

Quite a lot, it turns out — and not in the ways you'd expect. This post is about why I proposed removing it, how I made the case, what I put in place instead, and what happened after.

The Problem With Manual Gates

The argument for a manual gate is intuitive: it's a last line of defence. Before code hits production, a human reviews and approves it. That sounds like a good thing.

But in practice, manual gates create a specific set of problems that are easy to miss until you're looking for them:

They slow down critical fixes. The scenarios where you most need to deploy fast — a production incident, a data fix, a security patch — are exactly the scenarios where a manual gate adds the most delay. You need someone available, awake, with enough context to approve, while the incident is happening.

They create unclear commit bundles. When deployments are infrequent — because they require a manual step — multiple changes accumulate between releases. When something goes wrong, you're not reverting one change, you're reverting a bundle of them. Identifying root cause becomes harder. Rolling back a specific change becomes harder.

They breed false confidence. The person clicking approve rarely has enough context to catch a real problem. They're not re-running tests or reviewing every diff — they're approving based on trust in the process that already ran. If that trust is warranted, the gate is redundant. If it isn't, the gate is providing security theatre.

Real incidents have happened because of them. A production issue with duplicate records traced back to a toggle cleanup that was delayed waiting for a gate to be approved. Continuous Delivery would have caught it faster — the change would have gone through a smaller, faster cycle with less accumulation.

The key insight

Manual gates often feel safe but sometimes create bottlenecks without adding actual quality checks. The goal isn't to remove safety — it's to replace a manual checkpoint with real safeguards that actually catch problems automatically.

What I Proposed

The proposal was simple: remove the manual approval step from the CI/CD pipeline for PROD deployments, and replace it with a set of automated safeguards that are more reliable than a human click.

The key framing I used to make the case: this is not about bypassing safety — it's about replacing a fake safety mechanism with real ones. A human clicking "approve" on a pipeline they didn't run isn't a safety check. Automated tests, real-time monitoring, fast rollback capability, and feature toggles are.

The Case for It — Benefits

⚡ Accelerated Feedback Loops

Deploy faster and get real customer feedback sooner. Smaller, more frequent deployments mean faster learning and faster course-correction.

🚫 Reduced Manual Bottlenecks

Eliminate the dependency on a specific person being available at the right time. Deployments happen when the pipeline says they're ready.

🔐 Higher Confidence in Automation

When deployments are automated and reliable, trust in the CI/CD process grows. The pipeline becomes something the team relies on, not something they work around.

🧑‍💻 Increased Developer Productivity

Less time waiting means more time building. Engineers stop context-switching back to "is that deployment approved yet?" and stay focused on the next thing.

The Safety Measures That Replace the Gate

This was the critical part of the proposal. Removing the gate without replacing it with something real would be reckless. Here's what I put in place:

🚩
Feature Toggles
High-risk changes deploy behind a feature flag. The code is in production but the behaviour isn't active until the flag is turned on — independently, with monitoring in place. If something goes wrong, the flag goes off in seconds without a full rollback.
📊
Real-Time Monitoring and Alerting
Monitoring, logging, and alerting were already in place, but I tightened them before removing the gate. Error rates, latency spikes, and anomalous patterns trigger alerts immediately after any deployment — giving the same visibility the manual approver thought they were providing.
⏪
Automated Rollbacks
A reliable rollback plan — ability to revert to a stable version in minutes — is the real safety net. If a deployment causes a problem, rolling back is faster than waiting for a hotfix approval cycle.
🛡️
Production Failover
A backup path is maintained for high-impact failure scenarios. The gate removal doesn't reduce our ability to recover — it improves it by making the recovery path automated rather than dependent on human intervention.
✅
Enhanced Test Coverage
Critical workflows — the ones where a failure in production would be most damaging — are covered by unit, integration, and functional tests that halt the pipeline if they fail. The automated tests do the work the manual approver was supposed to do, but more reliably and at every commit.

How We Rolled It Out

1
Start with one service as a pilot
Rather than removing the gate across all services at once, I started with one — the one I had the most confidence in, monitored it closely for two weeks before expanding.
2
Introduce feature toggles first
Before removing the gate, we made sure any high-risk changes in that service were behind toggles. The gate removal only happened once toggles were in place and the team was comfortable using them.
3
Verify monitoring and rollback capability
I ran a simulated rollback drill — deployed a change, then rolled it back — to confirm the process worked as expected before relying on it in a real incident.
4
Train the team on the new workflow
The workflow change required the team to understand how to use feature toggles, how to read deployment alerts, and how to trigger a rollback. I ran a short session on this before go-live.
5
Expand to other services
After two stable weeks on the pilot, I expanded the approach. Each additional service followed the same pattern — toggles first, monitoring verified, rollback tested, then gate removed.

What Actually Happened

Deployments got faster. The anxiety around deployment days decreased — because deployments stopped being events and became routine. Smaller, more frequent releases meant that when something did go wrong, the blast radius was smaller and the root cause was easier to identify.

We haven't had a production incident caused by the gate removal. Incidents have happened — but the automated rollback capability that was built as a prerequisite caught problems faster than the old manual-gate process would have.

The outcome

Removing the manual gate didn't make deployments less safe. It made them more reliable — because we replaced a process that felt safe with tooling that actually is.

When a Manual Gate Is Still the Right Call

To be clear: there are scenarios where a manual gate makes sense. A major infrastructure change. A schema migration that can't be rolled back. A deployment with regulatory approval requirements. These deserve a human in the loop.

The problem isn't manual gates in principle — it's using them as a blanket default for every deployment, including routine application changes that are well-covered by automated tests. The goal is to reserve human review for the decisions that genuinely require it, and automate everything else.
